# AI Collaboration Standards

> 融合 Andrej Karpathy 的系统级编程思维与 VentoStack 工程实践。

## 1. 系统级思考 (Systems Thinking)

- **修改任何代码前**，先理解数据流、控制流和依赖关系
- **自顶向下阅读**：先读高层接口和类型定义，再深入实现
- **追踪数据流**：理解数据如何流入和流出系统
- **关注不变量**：找出代码中始终为真的条件
- **理解错误处理路径**：异常路径往往揭示设计意图

### 实践检查清单
- [ ] 是否理解了该模块在整体架构中的位置？
- [ ] 是否追踪了数据从入口到出口的完整路径？
- [ ] 是否识别了关键不变量和边界条件？
- [ ] 是否理解了错误处理策略和失败模式？

## 2. 第一性原理 (First Principles)

- **质疑每一个抽象层**：这个抽象真的必要吗？
- **不盲目遵循框架或模式**，从底层理解问题
- **正确性 > 性能 > 优雅**：先做对，再做快，最后做漂亮
- **简单优于复杂**：能不用设计模式就不用
- **显式优于隐式**：代码应该清楚地表达意图

### 决策框架
```
问题 → 底层原理是什么？ → 最简解决方案？ → 是否需要抽象？ → 验证正确性
```

## 3. 深度工作规范

- **长时间不间断专注于复杂问题**，避免上下文切换
- **理解问题的所有边缘情况后再动手**
- **修改前先完整阅读相关文件**，不凭猜测编码
- **遇到不确定时，先停下来思考**，而不是快速尝试

## 4. 代码修改原则

### 4.1 最小改动原则
- 只做必要的修改，不重构无关代码
- 一个 PR 只做一件事
- 不"顺便"修复不相关的问题

### 4.2 保持对称性
- 添加创建逻辑时，考虑对应的销毁/清理逻辑
- 添加配置时，考虑对应的验证和默认值
- 添加权限检查时，考虑对应的错误处理

### 4.3 不破坏现有契约
- 接口签名保持稳定
- 行为语义保持稳定
- 如需破坏契约，必须显式标记并更新所有调用方

### 4.4 测试先行
- 修改前先理解现有测试覆盖
- 修改后补充回归测试
- 安全关键路径必须有失败用例覆盖

## 5. 调试方法论

### 5.1 复现优先
- 确保能 100% 复现问题再开始修复
- 如果无法复现，先建立可复现的测试用例
- 记录复现步骤和环境信息

### 5.2 二分法定位
- 通过注释/条件快速缩小问题范围
- 使用版本控制回溯到问题引入点
- 隔离变量：一次只改变一个条件

### 5.3 理解根因
- 不满足于表面修复（symptom fixing）
- 追溯到根本原因（root cause）
- 问"为什么"直到到达系统边界

### 5.4 添加回归测试
- 确保问题不会再次发生
- 测试应该失败于修复前，通过于修复后
- 优先添加单元测试，必要时添加集成测试

## 6. 上下文管理

### 6.1 提供精确上下文
- 明确指定文件的相对路径
- 说明当前的目标和约束条件
- 提供足够的背景，但不要过度

### 6.2 迭代式开发
- 从小而具体的任务开始
- 验证每个步骤后再继续
- 及时回滚不工作的方向

### 6.3 明确约束
- 技术约束（Bun 优先、函数式优先等）
- 安全约束（信任边界、输入校验等）
- 性能约束（响应时间、资源限制等）

## 7. 代码审查思维

### 7.1 像攻击者一样思考
- 这段代码有什么漏洞？
- 输入校验是否充分？
- 边界条件是否处理？
- 并发场景是否安全？

### 7.2 像维护者一样思考
- 6 个月后我能理解这段代码吗？
- 命名是否清晰？
- 是否有足够的注释说明"为什么"？
- 依赖关系是否清晰？

### 7.3 像用户一样思考
- 这个 API 易用吗？
- 错误信息是否有帮助？
- 文档是否准确？
- 行为是否符合直觉？

### 7.4 局部化变更
- 修改的影响范围应该可控
- 避免"蝴蝶效应"式修改
- 保持模块的内聚性

## 8. 学习策略

### 8.1 技术深度
- 深入理解底层原理，而非表面 API
- 阅读优秀项目的源码
- 手写实现来验证理解

### 8.2 持续学习
- 关注技术演进，但不被 hype 驱动
- 建立个人的知识体系
- 教授他人来巩固理解

## 9. 质量准则

1. **正确性 > 性能 > 优雅**：先做对，再做快，最后做漂亮
2. **显式优于隐式**：代码应该清楚地表达意图
3. **简单优于复杂**：能不用设计模式就不用
4. **局部化变更**：修改的影响范围应该可控
5. **可测试性**：难以测试的代码往往设计有问题
6. **可观测性**：关键路径必须有日志、指标或追踪

## 10. 与 VentoStack 原则的融合

| VentoStack 原则 | AI 协作实践 |
|----------------|------------|
| Bun 优先 | 质疑每个 npm 包的必要性 |
| 函数式优先 | 优先纯函数，副作用显式标记 |
| 显式依赖 | 依赖关系必须可读、可跳转 |
| 编译期安全 | 类型必须完整，禁止 any |
| 默认安全 | 像攻击者一样审查代码 |
| 默认可审计 | 关键操作必须留下证据 |
| 后端先行 | 先稳定契约，再扩展前端 |
| 系统级思考 | 修改前理解架构全貌 |
| 第一性原理 | 质疑抽象，追求简单 |
| 正确性优先 | 测试先行，根因修复 |
